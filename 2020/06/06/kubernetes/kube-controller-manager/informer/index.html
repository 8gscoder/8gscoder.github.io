总览

index

其实就是线程安全的带读写锁的、带索引的map和slice

Delta-FIFO

Delta的意思是变化，而DeltaFIFO就是一个变化事件的Queue，底层是map/slice存储对象实现的一个FIFO Queue，并且再底层获取具体obj的是一个Store，就是上面的Index

Delta-FIFO的队列主要包含了Store和一个key的queue，注意，其实Queue中的元素逻辑上是key，而一个key可能有很多Delta事件，所以items其实是一个map[string]Deltas，注意是Deltas，带s的，即是一组Delta

这个队列入队是通过Add()、Delete()、Update()、Resync()，其实还有Replace()，Replace会把一个list全部入队，并设置initialPopulationCount参数，这个参数后面说

图片来源看结尾

我主要想看下Resync和HasSynced到底是什么

Resync

对list一下底层index Store存储的key，遍历并塞入到Queue里面的item，都添加一个SYNC的事件，以及去除掉Add之后又Delete的Delta（这个可能是提高效率吧，没考虑上层业务）

HasResynced

initialPopulationCount是在每次Replace的时候会设置，而Replace其实就是初始化将一个list塞入到Queue的时候，即从list-watch那边初始化list得到的一个objlist，而什么时候initialPopulationCount会减少？就是DeltaFIFO进行Pop的时候，Pop的时候回判断initialPopulationCount是不是不为0，如果是的话就initialPopulationCount–，这个判断的目的是，最起码FIFO是已经底层的Store一致了

ListerWatcher

listerwatcher是一个client-go的lister和watcher的分装，这里是直接合apiserver交互的工具

Reflector

reflector是一个反射器，他有store底层存储，这个Store是DeltaFIFO（注意DeltaFIFO自己里面有Thread-Safe-Map Indexer），有listerWatcher去list-watch apiserver，有指定监听对象的类型反射，有resync的period值。

注意总览图，Reflector是和APIServer最近的组件，他会List-Watch Apiserver

Run

List-Watch函数

List-watch会先sync-with一下list出来的obj，然后再watch，调用watchHandler函数进行对watch到的事件进行分类调用Store接口入队

Controller

这块的controller不是kube-controller-manager的controller，这里的controller是一个Reflector和DeltaFIFO的组合

controller就是包含了Reflector和Process的对象，他的调用Reflector的Run，进行watch APIServer，然后processLoop进行Pop Reflector的Delta。看起来有点像总览里面的左半部分informer

sharedInformer

informer的意思就是通知者，sharedInformer，share会很疑惑，思考如下：

  1. 谁share？share的人是sharedInformer嘛，细分一下就是这个informer的processor
  2. share谁？注释可以看到，是共享底层store，这里的store是本地缓存，不是etcd

sharedProcessor

就关注下核心的distribute和run

distribute就是在接收到obj之后疯狂调用lister.add()，这个在下面有介绍，至于是谁在distribute呢？这个distribute函数其实就是上层informer会调用传入新的事件进来的入口。

run的话，就是把lister run()/pop()起来，阻塞在stopCh，结束后会clode掉lister.addCh

processorListener

主要是run()/pop()/add()三个函数的实现，利用一个RingGrowing环形缓存队列（会自增长的），实现缓存，在add()和执行回调函数之间实现了一级缓存，从而实现上层回调函数执行业务逻辑（其实Best practices的做法是只做入workqueue操作）执行慢会阻塞sharedProcessor写addCh的阻塞问题。感觉设计的还是挺精巧的。

整体逻辑就是addCh接收入队事件，缓存中有的话，会不断的写入nextCh传给run处理，没有的话就阻塞。

Run

sharedIndexInformer.Run()，初始化Controller（其实是个DeltaFIFO的包装）之后，

HandleDeltas

这里就是所说的入口，也是DeltaFIFO的Pop的Process处理器，其实就是分发给监听器处理。，值得关注的是会校验一次从cache的store中对比下是否存在，而不是简单的判断Delta事件
