Install-APIS

在installAPIResource的时候，发现不是拿apiGroupInfo进行注册的，而是先构造了个APIGroupVersion的结构，然后调用APIGroupVersion.InstallREST，看下这块是如何做注册的。

把APIGroupInfo里面的几个参数透传给了APIGroupVersion，比较重要的是Scheme字段，作为了好几个字段，说明这Scheme有很大的作用。其实可以理解为APIGroupInfo，是对Group的描述，里面包含多版本的Version，而APIGroupVersion则是对指定gv的描述，里面包含了指定gv的版本，以及各种Creater构造器、Convertor转换器、Serializer序列化工具，Typer导出类型的工具、Defaulter填写默认值工具…

然后看下APIGroupVersion.InstallREST，通过installer.Install，返回了注册好到go-restful的websevice，以及注册后的apiResource，然后把这个apiResource作为discoveryHandler再注册到ws上，然后把ws注册到genericAPIServer传过来的go-restful contianer里面。over！

然后看下APIInstaller.Install吧，这是一个正儿八经的注册REST地方，而且返回了注册后的APIResource。

他主要是通过newWebService，在注册的时候，是把serializer转到WebService的Produces里面了。这个后续再看。需要看runtime包了。然后把path（这里的path其实是resource name）registerResourceHandlers到ws上。

具体看下registerResourceHandlers

这是一个600+行的函数。分开看吧

主要看下这块，ws.PUT().To.Doc.Param.Operation.Produce.Return.Reads.Writes链，主要是defaultVersionedObject这个的反序列，已经producedObject的序列化

而defaultVersionedObject的由来也很明显，是通过Scheme的Creator来的，然后通过反射而来的。

看完了上面的会发现，这块就是注册路由，依据不同包StorageProvider，各个资源在pkg/regitry下面是否有重写Store的Create，或者其他接口，例外最重要的是需要提供New和NewList这两个函数指针给Store，不然上层分装的New接口在正儿八经registerResourceHandlers的时候，没把结构体注册到restful的处理链上。
