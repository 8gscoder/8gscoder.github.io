考虑简单性，选deployment这个资源来看下链路。（core group下面的组织比较特殊，都是预留资源）

RESTStorageProvider/APIGroupInfo

每个pkg/registry/{groups}包下面都会有一个rest包，包里面会自己实现RESTStorageProvider这个Interface，这个Interface是一个==工厂类型==，这个工厂的生产函数是NewRESTStorage，这个函数只要传入APIResourceConfigSource和RESTOptionsGetter，就可以返回一个APIGroupInfo结构体，在再install-apis.md里面也有提在何时注册。下面就看下调用的链路吧。

RESTStorageProvider这是一个工厂结构体，生产APIGroupInfo

这是一个开关配置，这个配置是master包传进来的，主要是看一些gv是否开启

这是一个获取RESTOption工厂，GetRESTOptions可以返回一个RESTOption结构体，这个结构体内有后端存储配置、解码器还有一些其他

APIGroupInfo结构体如下

由名字可以知道，就是一个Group的信息表，里面看到会有Group的version相关的资料。

主要是

  1. PrioritizedVersions，保存了当前Group支持Version的优先级。

  2. VersionedResourcesStorageMap，保存当前k=version，v=mapof(key:resource，storage)，即保存了从version到资源，到资源的storage的映射关系

  3. Scheme，默认都会被复制legacy.Scheme，具体何时赋值，需要看Scheme那块。Scheme包含了这个group下面各种gv到type-name（这里的type是结构体名，也是resource的名称，Scheme用了反射保存结构体名作为映射）映射关系，已经不同version之间结构体如何进行convert

Storage/REST

看下deployment的api实现，以及Storage的定义

可以看出来：

一个Storage可以包含多个REST，一个REST==继承==了genericregistry.Store，而这个Store，是Kubernetes实现了一种==泛型结构==封装了对底层runtime.Object这个Interface的存储层读写的RESTful API。

genericregistry.Store实现了staging/src/k8s.io/apiserver/pkg/registry/generic/registry/rest/下面各种接口，比如Creator/Deletor，还有各种Strategy，genericregistry.Store其实可以理解是kubernetes项目尝试实现的泛型store，通过runtime.Object和函数类型的成员参数来巧妙的实现Golang“泛型”，避免多种api resource实现自己对Store层的操作。

说实话，看到这里store的实现，着实需要感叹一下kubernetes的精巧设计。感觉APIServer的核心，应该就是k8s.io/apiserver/pkg/registry/generic/registry/store.go就在这块了吧…

Store implements pkg/api/rest.StandardStorage. It’s intended to be embeddable and allows the consumer to implement any non-generic functions that are required. This object is intended to be copyable so that it can be used in different ways but share the same underlying behavior.

Store实现了pkg / api / rest.StandardStorage。 它旨在可嵌入（embeddable adj. 可嵌入的），并允许使用者实现所需的任何非泛型功能。 该对象旨在可复制，以便可以以不同的方式使用，但共享相同的基本行为。

All fields are required unless specified.

除非特定的几个，其他所有字段都是必填字段。其实这里的意思很容易明白，因为Golang在语法级别没有支持泛型，而如果你要在Golang中实现泛型，最起码你要把你定义的这个泛型实现最小功能化，就是他要支持“泛”的最基本功能，允许用的人不重写来覆盖他的函数，

The intended use of this type is embedding within a Kind specific RESTStorage implementation. This type provides CRUD semantics on a Kubelike resource, handling details like conflict detection with ResourceVersion and semantics. The RESTCreateStrategy, RESTUpdateStrategy, and RESTDeleteStrategy are generic across all backends, and encapsulate logic specific to the API.

Store的预期用途嵌入在Kind特定的RESTStorage实现中。 Store在Kubelike资源上提供CRUD语义，使用ResourceVersion和语义处理诸如冲突检测之类的细节。 RESTCreateStrategy，RESTUpdateStrategy和RESTDeleteStrategy在所有后端都是通用的，并且封装了特定于该API的逻辑。

仔细看注释，里面核心其实就是几个func构造函数，和几个处理Object的ObjectFunc，还有个最重要的

Storage DryRunnableStorage，这个是对底层etcd的封装

然后store还有一些列的成员函数，会在staging/src/k8s.io/apiserver/pkg/endpoints/installer.go的registerResourceHandlers时候，转换成RESTful API
